# CEX 跨所期现套利策略

本文记录第一版跨所期现套利自动化系统的策略边界、模拟盘规则和未来实盘差异。第一版只做模拟盘，不发送真实订单，但模型按真实跨所资金分布设计。

## 策略方向

### 买现货 + 空 U 本位永续

这是第一阶段主策略。系统在现货交易所买入现货，同时在永续合约交易所做空同等数量的 U 本位永续。

适用场景：

- 永续价格高于现货价格，存在正基差。
- 永续资金费率为正，空永续方预期收取资金费。
- 现货交易所有足够 USDT，合约交易所有足够保证金。

收益来源：

- 基差收敛收益。
- 资金费率收益。

主要风险：

- 合约保证金不足导致强平风险。
- 现货腿和合约腿无法同时成交，产生单边裸露。
- 资金费率转负，持仓从收钱变成付钱。
- 基差继续扩大，账面亏损触发风控。

### 卖出现货库存 + 多 U 本位永续

这是第二阶段策略，只允许使用已有现货库存，不做借币卖空、不做现货杠杆。

适用场景：

- 现货价格高于永续价格，存在负基差或反向基差。
- 永续资金费率为负，多永续方预期收取资金费。
- 现货交易所已经持有足够现货库存，合约交易所有足够保证金。

收益来源：

- 反向基差收敛收益。
- 负资金费率下，多永续获得的资金费。

主要限制：

- 系统不能为了执行该策略去借币。
- 现货库存不足时必须拒绝机会。
- 如果未来接真实账户余额，仍要保留手动模拟库存能力，方便模拟盘复盘和压力测试。

## 收益模型

统一使用以下公式判断机会：

```text
预期净收益 = 基差收益 + 预计资金费率收益 - 手续费 - 滑点 - 安全缓冲
```

字段说明：

- 基差收益：现货成交价和永续成交价之间的价差贡献。
- 预计资金费率收益：按当前资金费率和预计结算次数估算。
- 手续费：交易所明确收取的费用，按现货 taker 和合约 taker 费率分别计算。
- 滑点：不是手续费，而是盘口深度不足或价格移动导致的成交价偏移成本。
- 安全缓冲：防止模拟盘过度贴近理论收益，为行情延迟、撮合延迟和估算误差留余量。

第一版滑点使用固定比例估算，后续可升级为订单簿深度撮合。

### 瞬时机会和持仓观察

真实期现套利不只看开仓瞬间的价差，也要看持仓期间多期资金费率是否能覆盖成本。因此系统区分两类收益：

- 瞬时净收益：按当前基差和 1 期资金费率估算，用于判断是否可以立即模拟开仓。
- 持仓预期净收益：按配置的多期资金费率估算，用于发现暂时不适合开仓、但值得继续观察的 carry 候选。

机会状态分为：

- 可执行：瞬时净收益和收益率达到阈值，可进入模拟开仓。
- 观察：当前收益不足，但多期资金费率后预期为正，只进入监控列表，不自动开仓。
- 不可执行：瞬时和持仓预期都不划算，或行情、风控条件不满足。

观察状态不能直接下单。真实盘里这类机会要继续检查资金费率持续性、基差变化、盘口深度和账户资金分布。

## 模拟盘和真实盘区别

模拟盘：

- 成交价来自盘口估算。
- 买入使用 ask，卖出使用 bid。
- 手续费、滑点和安全缓冲按配置扣减。
- 每个交易所独立维护虚拟 USDT、现货余额、合约保证金、冻结资金和持仓。
- 模拟资金可手动配置，也可做同一交易所内现货账户和合约账户之间的 USDT 模拟划转。
- 模拟盘用于验证机会质量、收益核算、资金分布、风控和清仓流程。

真实盘：

- 成交价必须来自交易所订单成交回报。
- 收益必须使用真实成交均价、成交数量、真实手续费、真实资金费率和实际平仓结果。
- 真实盘必须处理部分成交、撤单失败、API 超时、重复回报、合约持仓模式和保证金模式。
- 真实收益可能显著偏离模拟盘，模拟盘结果不能直接当作真实收益。

## 资金模型

第一版使用每个交易所独立虚拟账户：

```text
Binance: USDT、现货余额、合约保证金、合约持仓
OKX:     USDT、现货余额、合约保证金、合约持仓
Bitget:  USDT、现货余额、合约保证金、合约持仓
```

这样做是因为跨所套利不能假设资金能瞬间跨交易所移动。机会执行前必须检查实际涉及的现货所和合约所资金是否足够。

模拟盘资金管理规则：

- 现货 USDT、合约 USDT 和现货库存按交易所分别维护，不能跨所直接共用。
- 合约最多使用 3 倍杠杆，模拟保证金占用按 `名义本金 / 杠杆倍数` 计算。
- 合约 USDT 不能被调低到低于已冻结保证金，否则会低估持仓风险。
- 第二阶段 `卖库存 + 多永续` 只使用模拟账户已有现货库存；库存不足时必须拒绝执行。
- 恢复默认资金会清空模拟持仓、熔断动作和熔断状态，只用于重新开始模拟，不代表真实账户清仓。
- 真实盘资金管理不能靠手动覆盖余额，必须读取交易所账户、持仓、冻结金额和资金划转回报。

## 熔断和清仓

熔断含义：

```text
停止开新仓 + 对已有套利组合生成紧急平仓动作
```

触发条件：

- 单笔亏损超过阈值。
- 日亏损超过阈值。
- 单所余额不足或保证金不足。
- 行情延迟或数据失真。
- 单腿成交失败。
- 交易所 API 异常。

清仓原则：

- 清仓不能盲目市价全平。
- 优先保持两腿对冲关系，避免只平一边制造更大的裸露风险。
- 如果单腿失败，只成交的一边要立即反向平掉。
- 如果两边成交数量不一致，保留较小数量的对冲部分，剩余裸露部分优先平掉。
- 如果某个交易所不可访问，系统应进入紧急模式并提示人工介入。

## 自动开关仓和数据记录

第一版自动化默认关闭，由页面手动启用：

- 自动开仓只处理 `可执行` 状态的机会，并要求收益率达到配置阈值。
- 同一 `交易对 + 方向 + 现货所 + 合约所` 组合不会重复自动开仓。
- 自动平仓条件第一版支持最大持仓时间，以及预计收益率达到平仓阈值；`0` 表示关闭该条件。
- 自动逻辑每次循环先平仓再开仓，优先释放保证金和仓位上限。
- 当前自动平仓仍使用模拟盘估算收益，不等同真实成交利润。

机会记录：

- 只记录扣除手续费、滑点和安全缓冲后的正净收益机会；无利润候选只在实时扫描中展示，不进入历史记录。
- 有利润机会会记录首次出现时间、最近出现时间、出现次数、最近收益和历史最佳收益。
- 自动开仓失败会在机会记录中保留拒绝原因，例如余额不足、库存不足、保证金不足。
- 自动交易记录区分开仓和平仓动作，记录触发原因、交易组合、预计利润和收益率。

持久化规则：

- 自动化开关、自动开仓/平仓开关、开仓收益率、平仓阈值、最大持仓时间和仓位上限保存到 MySQL。
- 模拟盘资金账户保存到 MySQL，包括每个交易所的现货 USDT、合约 USDT、冻结保证金、现货库存和合约持仓。
- 模拟持仓也保存到 MySQL，避免服务重启后只恢复冻结资金却丢失对应套利组合。
- 手动改资金、资金划转、重置账户、手动开平仓、自动开平仓后都会保存最新模拟账户和持仓快照。
- 有利润机会记录保存到 MySQL，服务重启后继续保留最近机会和出现次数。
- 自动开仓、自动平仓流水保存到 MySQL，每条流水保留触发原因、组合、方向、利润和收益率。
- 自动累计利润、平均利润和胜率从自动平仓流水重算，避免服务重启后统计归零。
- `SIMULATION_ONLY=true` 调试模式不依赖数据库，只保留内存态；要验证持久化必须使用正常 MySQL 启动方式。

统计指标：

- 自动开仓次数。
- 自动平仓次数。
- 每次自动平仓利润。
- 自动累计利润。
- 自动平均利润。
- 胜率：`盈利平仓次数 / 自动平仓次数`。

这些数据用于后续优化策略阈值、持仓时长、币种池、资金分配和风控参数。

## 第一版实现边界

- 覆盖 Binance、OKX、Bitget。
- 支持任意现货交易所 x 任意 U 本位永续合约交易所组合，包括同所期现。
- 交易对使用自定义白名单，默认加入 Top20 主流 USDT 交易对；前端支持手动新增 `XXXUSDT` 形式的交易对。
- 第一版只做模拟盘，不真实下单。
- 反向策略只使用已有现货库存，不借币、不做现货杠杆。
